[edit] Terminology

[edit] Patch

A patch could be considered extra metadata to one or more commits. A patch represents a certain functionality change in the code. It holds information about why the change was done, what functionality it adds, and points to one or several commits that combined make up the functionality change. Every patch has an unique ID number which is prefixed PID. Commits are coupled to a patch by matching the commit title and the name of the git it modifies. The mapping data is entered by users in the "Review links" tab at the bottom of the patch dialog. Example of a patch: PID001234

[edit] Commit

A Commit is a commit as created by git commit. The commit data displayed in patchtool is extracted from our repository every 4 hours.

[edit] Assets

This is data that helps to describe what systems the patch is suitable for. Example: If a patch is only needed on the Japanese market, it should be marked with "Region: Japan". The purpose of this is that when we create new systems and products, we want to be able to compose these from existing patches and the asset data greatly helps to find the suitable scope. Especially for ODM projects the scope can be quite different compared to the in-house lead projects, so this info is necessary.

[edit] Tags in a patch

The "Tags" in a patch is a set of random strings which are used in e.g. filtered lists. A common usage of tags is to filter out patches that are obsolete for a specific android version (example: "obsolete/android-4.2_r1"). Currently, the tags are used in many ways, by teams and developers. The "freedom" for anyone to invent tags has resulted in a very large number of tags in PatchTool. For this reason, a set of documented "official" tags have been defined in Admin -> Tags. The idea is to reduce the number of tags by encouraging developers and teams to choose from a set of predefined and documented tags.

  • Go through "official" tags first and review the descriptions to see if there is a suitable tag already.
  • Use lower-case only and avoid special characters (if you see no other way than defining your own tag).

[edit] Contribution

This is used by the contribution officer during the contribution process. A contribution contains some metadata about a contribution and links to review links in Gerrit (both @ Google and @ SEMC). Every contribution has an unique ID which is prefixed CID.

[edit] Bringup - commits

The Commit bringup tool lists all commits that are available on the reference branch, but not on the bringup branch excluding those which has been filtered out. The bringup tool maps on commit titles and gitname which means that its important to keep the same title. There will also be a mismatch if a git has been split into several new gits.

The commit sequence numbers come from the ordered list of commits (git log) on the reference branch. Lowest number = added most recently. Highest number = added first. When bringing up a new version, cherry-pick the commits with high sequence number (old) first if there are dependences between the commits. When reviewing the dates, please note that AuthorDate and CommitDate may be different. (git log --pretty=fuller).

[edit] Bringup workflow

When you want to do a bringup, you can use the bringup tool to track the progress.

[edit] Preparing the bringup view

1) Click the Bringup-commits tool in PatchTool (or go to this link: PatchTool bringup)

2a) Select "Bringup project", it has a list of official bringup projects that have been configured by MBUT team.

2b) If the bringup project you are working on does not exist in the "Bringup project" list, you may enter your own source and destination branch:

  1. Select "Reference manifest", this is the "source" branch e.g. "ginger-mogami"
  2. Select "Bringup manifest", this is the "target" branch e.g. "ics-common"

3) (Optional) Set additional filters, for example "Sub Distribution" to your team's sub distribution.

4) Click on "Commit sequence" and make sure the sort order is highest number first.

5) Right click on "Patch ID" column and select "Group by Patch ID"

6) Now you can see a list of missing commits, this list is based on titles so it important not to change the title when you cherry-pick!

[edit] Doing the bringup

Work the bringup list from top down, for each commit and patch, take an action based on the below decision instructions: 

1) If the commit does not have a patch mapped to it: 
      a. If it belongs to the same functionality change as an existing patch -> Map the commit to the patch.
      b. else -> Create a new patch for it and enter appropriate patch info.
(Note that this should ideally have been made proactively before the bringup as part of regular development routines)

2) If the patch needs to be investigated further before applying or obsoleting it -> mark the patch with suitable "investigate/" tag.
         For example in Jellybean MR2 bringup: "investigate/android-4.3_r1". See info about patch tags [here].

3) If the commit should be applied to bringup target branch:
      a) Check if there are other commits that should be squashed with it, if so, follow the squash instructions below.
      b) Cherry-pick the commit according to instructions below. If you have done this correctly, the commit will disappear from the bringup list within 4 hours.

4) If the commit should not be applied to the bringup target branch, take one of the following decisions:
      a) If the patch is completely obsolete, meaning that none of its commits should never be applied in
         any following bringup -> set the patch to obsolete by changing its category value to "Obsolete".
      b) If the patch is obsolete from a specific android version and onwards -> mark the patch with suitable "obsolete/" tag.
         For example in Jellybean MR2 bringup: "obsolete/android-4.3_r1". See info about patch tags [here].
      c) If a specific commit should not be applied, but its patch is still not obsolete, for example when one of the
         commits in a patch is no longer needed, but the others are -> Check the “Not part of gap” checkbox and enter
         a comment that motivates why the commit is not applicable.
      d) If a commit needs to be rewritten completely so that cherry-picking is not suitable -> [follow these instructions]

[edit] Example: Cherry-picking two commits 

In this example you want to cherry-pick c23a9a2a6c884308b8cdebd7189b2799762a8968 and then 4d5940acebb35d9cb1e2f1dd0e8f00be5a7cd66a:

git cherry-pick -x c23a9a2a6c884308b8cdebd7189b2799762a8968
git mergetool -t kdiff3
git commit
git cherry-pick -n 4d5940acebb35d9cb1e2f1dd0e8f00be5a7cd66a
git mergetool -t kdiff3
git commit --amend #Don't forget "Squashed-With:" and also a "(cherry picked from commit 4d5940acebb35d9cb1e2f1dd0e8f00be5a7cd66a)"!

[edit] Squashing commits during a bringup

If you squash commits during bringup (if you have several commits in one patch) you must add a "Squashed-With: <title>" as a git footer. PatchTool uses this info to understand that the listed commits are now applied to the bringup target branch. Example: 

commit 19b628e52e1e04e5b1f4ca94e60f6a755708c2ce
Author: Mikael Gullstrand <mikael.gullstrand.x@sonyericsson.com>
Date:   Mon Nov 15 14:09:13 2010 +0100

   Native libs are removed.
   
   This is a squash of three commits to make sure that native
   libs are kept and the keep the performance up.
   (cherry picked from commit 94614c9314e1b2cb96962e4c67c94abd2de36544)
   (cherry picked from commit 3621798362136217893612879362178936218793)
 ...
   
   Change-Id: I5d98d5fcb321b0ca1fc10a39cc2934a86608b623
   Squashed-With: Improve startup performance.
   Squashed-With: Additional fix for Improve startup performance.

NOTE: Do not wrap the text after 'Squashed-With' - it is permitted to break the 72 char limit.

[edit] When a change is still needed but is not cherry-picked

Sometimes a fix or change is still needed but the code needs to be modified so significantly that cherry-picking or squashing is not appropriate. In other words, the patch is still needed, but the commits are not valid for the bringup and needs to be rewritten. In that case do the following: 

1) Create a new commit and submit it to the bringup target branch.
2) Map the new commit to the patch it belongs to in PatchTool.
3) In the bringup view, check the "Not part of gap" checkbox for the old commit that is no longer valid.
4) Write a suitable explanation in the "Reason for not part of gap" column. (example: Commit rewritten in SHA1").

[edit] Changing commit title during a bringup

If you want to create a better commit title during a bringup you can do this with by adding the original title as a "Squashed-With:" footer. NOTE you must then also map the new gerrit number to the patch! 

commit 12345567890338e52e1e04e5b1f4ca94e60f6a755708c2ce
Author: Marvin The Robot <marvin.robot@sonyericsson.com>
Date:   Mon Nov 15 14:09:13 2010 +0100

   Correct synchronization in WindowManagerService
   
   There was an ugly bug hidden here.
   
   (cherry picked from commit 94614c9314e1b2cb96962e4c67c94abd2de36544)
 ...
   
   Change-Id: I5d98d5fcbc2eb0ca1fc10a39cc2934a86608b623
   Squashed-With: Old ugly commit title

[edit] Automatic trying to apply a list of commits

This is for "advanced" usage. A script will iterate through all commits in a list and run a verify command on each cherry-pick (e.g. build) and if everything was ok it will continue with next, otherwise it will skip the current commit.

Download a list to be cherry-picked by pressing "Download cherry-pick file". Make sure you have the scripts from semctools/bringup-tools in your PATH: 

cd ~
git clone git://review.sonyericsson.net/semctools/bringup-tools
export PATH=$PATH:~/bringup-tools/scripts

Navigate to correct git and start the automatic cherry-pick script: 

cd ~/my_repos/ics-riogrande
source build/envsetup.sh
lunch nypon-userdebug
cherry-pick-list-forwards.sh --autoskip --printfailed --printapplied --verifycommand compile-verify.sh cherry-picks.txt

Note that you might want to modify the compile-verify.sh to fit your needs (run automatic test cases or whatever).

[edit] Bringup – manifests

The main purpose of this panel is to track differences and changes in manifests. Given two manifest names, it is possible to generate the following reports:

  • list gits which are available only in one or the other manifest.
  • All gits union of gits.
  • List gits where the underlying SHA1 revision is different. (regardless if the manifest contains a tag/branch/SHA1 etc)

Metadata (notes and state ID) is maintained by the users, while the rest of the information is retrieved directly from git.

  • Click on "notes" to edit a comment about a git in the given combination of reference manifest and bringup manifest.
  • Click on the git name to see a list of available notes made on all manifest bringups. The notes field is editable in this dialog.
  • Click on "State" to set the progress of the manifest bringup.

Changes are logged in "History" panel.

Examples:

  • http://patch.sonyericsson.net/#manifests,listtype=0,refmanifest=platform/manifest:jb-lagan-1.0.1,manifest=platform/manifest:jb-lagan-1.1,search=state:Open

In this example, the tool will list the missing gits in jb-lagan-1.1 (when bringing up the scope in jb-lagan-1.0.1). The git is either actually missing from the bringup manifest or it is not yet marked as obsolete (Done) by an user:

  • If the git is added to the manifest it will disappear from the gap list automatically.
  • Users may use the state to exclude it from the "todo" list. (if the git is obsolete in the new version).

[edit] Searching in Patchtool

[edit] Searching with the filter field

In many of the tools there is a field named "Filter". Typing a string in this field will search all fields and filter out the commits/patches that does not meet the criteria. If you want to search a specific metadata field you can do this with fieldname:phrase, a ! in front means that the field should not contain the phrase. The following example will filter the result to only contain commits and patches that are in the git frameworks/base and that does not have a tag containing the string "todo" 

 gitname:frameworks/base !patch_tags:todo

[edit] URL link search

If you know a gerrit number and want the patch for this you can link directly to this by typing:

http://patch.sonyericsson.net/#change,70080

If you know a PIDs id you can search with:

http://patch.sonyericsson.net/#patch,607

If you know a CIS id you can search with:

http://patch.sonyericsson.net/#cid,547

If you know a SEMC gerrit number and want to find a contributon for this:

http://patch.sonyericsson.net/#semcchange,82484

If you know a Google gerrit number and want to find a contributon for this:

http://patch.sonyericsson.net/#googlechange,21909

[edit] Browser extensions

With this browser plugin you can instantly access the corresponding patch when reviewing a commit in gerrit. The color of the icon shows if the commit is registered in PatchTool. Download the plugin from the PatchTool home page here: http://patch.sonyericsson.net/#home

  • When visiting review.source.android.com and click on a review link you get an icon in the address field. If you click it you are directed to patch.sonyericsson.net and the corresponding contribution is opened.
  • When visiting review.sonyericsson.net and click on a review link you get an icon in the address field. This is red if there is no patch for it and green otherwise. If you click it you are directed to patch.sonyericsson.net and the corresponding patch is opened (or you get a question if you like to create a new one)

Gerrit dashboard for other users

After gerrit upgrade you lost the links to other users dashboards, if you want to see other users reviews and latest merges you can modify the link gerrit creates to a users open issues. 

1. Install the Firefox Redirector Plugin
2. Make a regexp link redirect:
Match: (.*?)#/q/owner:([^:]*?)\+status:open,n,z
Replace: $1#/dashboard/?title=$2&Owned+by=owner:$2+status:open,n,z&Reviewer=reviewer:$2+status:open,n,z&Recently closed=(reviewer:$2+or+owner:$2+)+status:closed+limit:15,n,z

[edit] Dealing with split and renamed gits

For example in jellybean, the frameworks/base is split into several gits. In Admin -> Gits it is possible to assign an alias for any git. Example:

  • platform/frameworks/av -> platform/frameworks/base
  • platform/frameworks/native -> platform/frameworks/base

Given this alias configuration, patchtool will take the commits in frameworks/av and frameworks/native into account when computing the delta for frameworks/base in bring up tab. This is also useful when gits change name between cookies. The configuration is done in admin/git manager.
